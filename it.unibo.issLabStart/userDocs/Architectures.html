Appunti

  1. Parto dal problema a dalla analisi con l'obiettivo di formalizzare TestPlans
  2. Parlo di sistemi software e quindi di componenti e loro organizzazione
  3. Capisco che ci sono bias nel modo di pensare e che un problema (o la logica/esperienza) suggerisce forme di organizzazione non pienamente catturate dai linguaggi di programmazione disponibili
  4. Occorre parlare di struttura, interazione e comportamento delle parti (dei componenti) e del tutto (del sistema) riconoscendo che non sempre le propreità del tutto sono riconducibili alle proprietà delle parti. Il modo di interconnettere e di interoperare i componenti fa 'emergere' nuove proprietà.
  5. I sistemi software sono utili perchè svolgono funzioni. Lo possono fare 'su richiesta' di enti esterni a loro oppure perchè capaci di comportamento autonomo, come ad esempio il sistema di controllo di un robot.
  6. I sistemi vengon progettati e costruiti da noi, e potrebbero possedere anche proprietà 'auto-organizzanti' ed essere capaci di replicazione. Ricerche nel campo della AI e della Biologia cercano di capire come sistemi possano auto-costruirsi in base alle leggi fisiche conosciute e a 'eventi' capaci di creare strutture composte (complesse) a partire da parti più semplici.
    There are no separate systems. The world is a continuum. Where to draw a boundary around a system depends on the purpose of the discussion. Donella H. Meadows B8
  7. A noi interessa costruire sistemi capaci di soddisfare requisiti epressi in linguaggio naturale e organizzati in modo da potere essere (facilmente) modificati (estesi), manutenuti e deployed. Un ruolo fondamentale è svolto dalla Architettura del sistema in un ecosistema di software development in rapida evoluzione.
    An architect is expected to define the architecture decisions and design principles used to guide technology decisions within a software development team.
  8. La definizione della architettura più appropriata di un sistema è sempre legata ad una accurata analisi del problema ed è sempre il risulato di un compromesso.
    • Architecture is about the important stuff ... whatever that is (Ralph Johson) 1
    • Everything in software architecture is a trade-off. (First law of Software Architecture) 18
    • Why is more important than how. (Second law of Software Architecture) 18
    • The architecture for a software system arises from the combinations of requriments and additional concerns (Auditability, Performance, Security, Legality, Scalability ...) each protected by fitness functions B10 (objective functions used to summarize how close a design solution is to achieving the set aims)
    • All architecture become iterative becuase of unknown of unknowns. What we need today is an Evolutionary Architecture. 15 The evolvability is a meta-characteristic that protects all the other architectural characteristics, to ensure they don't erode B6.
    • An Evolutionary Architecture supports guided, incremental changes across multiple dimensions B25 including -ilities B 2 and broader (Technical, Data, Security, Operational B9)
    • The software development ecosystem (tools, frameworks, prectices, ..) forms a dynamic equilibrium B4 - much like a biological system.
    • The Continuos Delivery and DevOps movements added a new factor in the dynamic equilibrium. B73
    • To build evolvable software systems, architets must think beyond just the technical architecture.

    From Gradle User Manual
    1. a build essentially configures a set of tasks and wires them together — based on their dependencies — to create that DAG.
      Once the task graph has been created, Gradle determines which tasks need to be run in which order and then proceeds to execute them.
    2. Gradle evaluates and executes build scripts in three phases (that form Gradle’s Build Lifecycle): Initialization, Configuration, Execution
    3. Gradle provides several mechanisms that allow you to extend it:
      • Custom task types (It’s typically best to put the source file for a custom task type in the buildSrc directory)
      • Custom task actions
      • Extra properties
    4. well-designed build scripts describe what steps are needed to build the software, not how those steps should do the work. That’s a job for custom task types and plugins.
